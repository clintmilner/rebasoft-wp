Introduction

Auditor is designed for use in network and security operations within enterprises and service providers. Its core capability is to report on network assets, providing the ability to locate users and devices as they connect to the network. The use of this core information enables Rebasoft to deliver significant benefits beyond being a repository of information.

With the network as the index, Auditor enables:

  • faster mean-time-to-fix problems
  • improved capacity and change management
  • improved security asset management
  • platform for managing VoIP
  • and more

Auditor goes beyond port / device tracking utilities in relating real-world information to the network. After all it is not an IP address that rings a helpdesk complaining of a problem.

The system uses your network switches, routers and firewalls as the primary data source, regularly collecting information using SNMP and DNS. We call this the "network index".

Depending on the role of the individual using Auditor, the indexed information can be presented in different ways depending on how the user wants to use the system

Auditor's data can be used to help in problem management, reducing the time spend in fixing user issues by getting to the root of a problem faster, empowering the helpdesk

Auditor's data can be used to reduce time and effort in changing the network. Building change plans based on real-time information on available resources and where users are connected.

Auditor's real-time support for user connections to the network allows fast isolation of devices, and supports quick evaluation of end device threat

One of the most powerful functions is to tag network elements with real-world information automatically. So, a userid could have a real person (firstname, surname), a contact phone/ email address, a manager, a department.

Reporting can now be not just about the network, but who's using it and the impact they are having on it. Organizations can start to understand where the threats/miss-usage actually occur, and record over time for Audit purposes. This leads to less downtime, better return on infrastructure investment, improved productivity and hence reduced costs.

Key features of Auditor include:

  • Automatically build a complete index of the network including infrastructure and connected devices

  • Relate real-world information to discovered elements

  • Keep a history of resources by user over time

  • Track asset utilization

  • Build capacity / change plans

  • Instant Active Tests to verify remote IP and ports

  • Watch list alerting for specific devices accessing the network

  • White list alerting about new devices accessing the network in real-time

  • Powerful, fast searches for information

  • Extensive integration to external applications

  • Open application with full URL control

  • Low system resource and overhead – with extensive configuration options

  • Simple and fast deployment

    System parameters

    Auditor relies on SNMP polling device CAM tables as part of information collection. The software has been extensively tested in large networks and against different device types and vendors. System parameters are set as a result of the testing and in line with system recommendations below.

    Experience shows that it takes around 1 minute for a device to respond to poll of the CAM table. The system default is for the polling cycle to be completed every 30 minutes. The objective therefore is to set the number of concurrent poller threads to be sufficient to complete to information collection and processing within the allotted time.

    We recommend that 1 thread be allowed for each 10 devices to be polled. The installation default is for 20 threads, this would allow support of up to 200 devices (with between 5000 and 8000 interfaces). The thread count can be increased subject to network and server load considerations.

    License Administration

    Auditor's license is deeply embedded within the application. License information is constantly evaluated as devices are polled for SNMP information. The active interfaces for a device are updated and added to the total count for all devices polled; so from hour to hour and day to day, the number of active interfaces will vary.

    When the number of active interfaces polled exceeds the licensed number of interfaces, the data for interfaces in excess of the license maximum are not stored in the database for that polling period. If, subsequently, the total number (including the extra ones) falls below the license level, data will once again be written to the database). Polling of interfaces from a device happens in the order the device presents the data to Auditor, this varies by vendor. Devices are polled in sequence, typically every 30 minutes.

    If switches or other devices are no longer required the license can be recycled by pressing  next to the device in question.

    Licensing rational

    Auditor core is licensed by the total number of up interfaces. The total is calculated by adding all the up interfaces on devices polled by Auditor.

    Note that once an interface is seen up it is counted in the total until the device licenses are recycled.

    Capacity planning module is a separate license option based on the number of Auditor core interface license level.

    Searches

    Auditor's powerful search engine allows users to search for items in the database. Most users will use the search as way to quickly navigate to the point of interest. Searching can be on full or partial matches to the string entered in the search box.

    Search accuracy can be improved by specifying a more complete string. For example, searching on "10.100.1" will bring up all the matching records with the string – in this case matching IP addresses and devices.

    Searches return results in for MAC, User ID, IP address, Hostnames, Device details, Interfaces and VLANS:

    The search results table lists the possible matches. Clicking items on the list allows a user to navigate to the relevant forensics report.



    Only when you select a search result does Auditor query the database (default 8 days) to see when the item occurred. Information displayed in timelines allows users to view longer or short periods of time as required. Using this method Auditor, for example, does require users to know when someone was on the network to see if they've ever logged in.

    Events

    Auditor alerts users and logs these events to the web console or, optionally, one or more syslog servers.

    Events are classified by type (System, User, Watchlist, Whitelist and device).

    Clicking any of the hyperlinks will filter the event list to display the relevant information.

    Events can be deleted, acknowledged or un-acknowledged. These actions do not affect any events that have already been forwarded to any external syslog received configured in Syslog settings for Alert settings.

    User forensics

    User forensics can be used to find the IP and hence the MAC history of a UserID. User forensics will show the last seen connection for the user – relating UserID, to IP address, to MAC Address to Device and interface. Device and Interface details will only be shown if Auditor is collecting information from the device that the users is connected to.

    User forensics will be populated only if has been correctly configured, and Auditor has had sufficient time to start collecting user data from LDAP or security event log sources.

    Hostname forensics

    Auditor has the ability to report on the hostname history for any IP address indexed by Auditor. Each IP address indexed by Auditor is added to a collection list and Auditor attempts to resolve the hostname via DNS. If a DNS entry is found, Auditor records this entry in its database. Administrators can control the frequency of DNS lookup, retention period and concurrent lookups via the Database Settings . When a DNS lookup successfully returns a hostname, the data is stored in Auditors database.

    DNS overview allows a top-level view of hostname assignments allowing, at a glance, a view of the most active DNS assignments:

    Administrators can click on a hostname to view the Hostname forensics report. This report details the IP assignment history for the selected hostname.

    Watch lists can be added via 2 methods. From the MAC forensics screen by clicking the "Add Watch entry" link:

    Or via the Watch list admin drop down:

    Watch lists are configured in the following screen:

    1.     Add device (plus a description) to the watch list.

    2.     Once the watch list device is added, decide on an alerting setting if required.

    White list

    While list is selectable by site, and is a facility to generate alerts on new devices accessing the network at that site. For example if you have a locked down standard build for a remote location, you can use Auditor  to alert on any device not previously seen connecting to the network at that site. White list alerts are reported to the Site overview screen, the events system and, optionally, a syslog receiver.

    White lists are configured by accessing the admin selector:

    White list settings provide the ability to start, stop and reset White list monitoring. If a configured site has one or more devices being polled by Auditor , monitoring can be activated by clicking the "Monitor" button.

    If an asset system already records the MAC address of devices, these can be manually added to Auditor  via the "Add" button.

    Once monitoring has been activated new devices seen are alerted to the event subsystem, where the device, interface and MAC address are shown. Users can click the MAC address to view MAC forensics

    White lists, once set up can be changed and updated via the White list administration dialog:

    Newly discovered MAC addresses can be automatically included in the White list by clicking the "Reset" button. Users can also add a list of MAC addresses by clicking the "Add" button.

    MAC addresses can be added singly to the White list by selecting them from the "new" list

    Individual MAC addresses can be added by clicking "Add to White list"

    For roaming users, a Global Whitelist override is provided. Any MAC address in the global list will not generate an alert if newly discovered at a monitored site:
